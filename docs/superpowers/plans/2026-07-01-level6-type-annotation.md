- [ ] **步骤 1.2：实现最长前缀匹配函数**

新增函数 `前缀匹配`，接收字符串和关键字列表，返回匹配到的关键字和剩余字符串。匹配策略：从最长关键字开始尝试。

```duan
段 前缀匹配 接收 s, kws：
  设 i 为 0。
  设 best 为 ""。
  设 best_len 为 0。
  当 i 小于 列表长度(kws)：
    设 kw 为 列表获取(kws, i)。
    设 kw_len 为 字符串长度(kw)。
    如果 kw_len 大于 best_len：
      如果 截取(s, 0, kw_len) 等于 kw：
        设 best 为 kw。
        设 best_len 为 kw_len。
      结束。
    结束。
    设 i 为 i 加 1。
  结束。
  如果 best_len 大于 0：
    返回 列表创建(best, 截取(s, best_len, 字符串长度(s))).
  结束。
  返回 列表创建("", s).
结束。
```

- [ ] **步骤 1.3：修改 `段 词` 函数**

将原本的贪心读取改为：先读取连续中文字符+字母+数字到下个非字母字符，然后用 `前缀匹配` 从最长关键字开始匹配。如果匹配成功，返回 KW token，剩余部分递归调用 `词` 继续处理。

注意：`词` 函数的接口不变（返回 `建('KW', kw)` 或 `建('ID', s)`），但内部实现改为前缀匹配+递归。

```duan
段 词 接收 src, p, n, s：
  # 先尽量读取完整的标识符/关键字
  当 p 小于 n：
    设 c 为 字符串获取(src, p)。
    设 是字母 为 假。
    如果 c 等于 "_"：设 是字母 为 真。结束。
    如果 c 大于等于 "a" 且 c 小于等于 "z"：设 是字母 为 真。结束。
    如果 c 大于等于 "A" 且 c 小于等于 "Z"：设 是字母 为 真。结束。
    # 中文字符
    如果 c 大于等于 "一"：设 是字母 为 真。结束。
    # 数字也可以作为标识符的一部分（不能开头）
    如果 c 大于等于 "0" 且 c 小于等于 "9" 且 s 不等于 ""：设 是字母 为 真。结束。
    如果 是字母 等于 假：跳出。结束。
    设 s 为 s 加 c。
    设 p 为 p 加 1。
  结束。
  如果 s 等于 ""：返回 建("EOF", "").结束。
  # 最长前缀匹配（仅在 s 包含中文时才能匹配中文关键字）
  设 kws 为 关键字列表()。
  设 匹配结果 为 前缀匹配(s, kws)。
  设 kw 为 列表获取(匹配结果, 0)。
  设 剩余 为 列表获取(匹配结果, 1)。
  如果 kw 等于 ""：
    # 未匹配到关键字，整个字符串是标识符
    返回 建("ID", s).
  否则：
    # 匹配到关键字，剩余部分如果是非空，也要继续处理
    # 但这里只返回当前关键字 token
    # 注意：剩余内容在本轮不做处理——因为 扫 函数会重新读取
    返回 建("KW", kw).
  结束。
结束。
```

关键说明：当 `设x为10` 输入时，`词` 函数会读到整个 `设x为10`，然后 `前缀匹配` 匹配到 `设`（因为 `设x` 不是关键字），剩余 `x为10`。但由于 `词` 的返回机制，这里需要一种方式告诉调用方"还有剩余内容需要处理"。实际上更好的方案是：`扫` 函数每轮只读一个 token，但 `词` 可能匹配到整个 `设x为10` 后返回 `设`。剩余的 `x为10` 需要在下一次循环中处理。

这里需要修改 `词` 的返回机制：除了返回 token 和新的处理位置 p 之外，还需要返回下一个处理位置。但当前 `词` 只返回一个 token。

**更好的方案**：在 `扫` 函数中配合。`扫` 调用 `词` 获得 token 后，不把 p 跳转到整个字符串末尾，而是只跳转到匹配到的关键字/标识符的末尾。对于 `设x为10`，`词` 读到整个字符串，`前缀匹配` 匹配到 `设`(长度1)，所以 `扫` 把 p 设为 p+1，然后下一轮 `扫` 再次调用 `词`，此时从 `x为10` 重新开始读取。

所以 `词` 需要返回两个值：(token, consumed_len) 而不是 (token)。但当前设计只返回一个建(t,v)。

**简化方案**：保持 `词` 返回一个 token，但修改 `扫` 对 `词` 的处理逻辑——`扫` 在调用 `词` 后不直接 p += len(token_value)，而是根据匹配情况确定实际消耗的长度。

- [ ] **步骤 1.4：修改 `段 扫` 函数**

修改 `扫` 函数：
1. 在扫描每一行时，调用 `统计缩进` 记录行首缩进
2. 在行与行之间，比较缩进变化并插入 INDENT/DEDENT token
3. 对空格/制表符的处理改为行首统计（而不是直接跳过）
4. 新增运算符符号映射（`==` → EQ, `!=` → NEQ 等）

关键：`扫` 函数需要逐行处理，而不是逐字符。这样能正确处理换行和缩进。

```duan
段 统计缩进 接收 line：
  设 缩进数 为 0。
  设 i 为 0。
  设 长度 为 字符串长度(line)。
  当 i 小于 长度 且 字符串获取(line, i) 等于 " "：
    设 缩进数 为 缩进数 加 1。
    设 i 为 i 加 1。
  结束。
  返回 缩进数。
结束。
```

- [ ] **步骤 1.5：添加运算符符号/英文别名支持**

在 `扫` 函数中添加对符号运算符的处理。当遇到 `+`、`-`、`*`、`/`、`%`、`==`、`!=`、`<`、`>`、`<=`、`>=`、`&&`、`||`、`!`、`and`、`or`、`not`、`=` 等符号时，映射到对应的内部 token。

```duan
# 在 扫 函数中添加
如果 c 等于 "+"：
  列表追加(toks, 建("ADD", "加"))。
  设 p 为 p 加 1。
  设 已处理 为 真。
结束。
如果 c 等于 "-"：
  # 检查是否是 ->
  如果 p 加 1 小于 n 且 字符串获取(src, p 加 1) 等于 ">"：
    列表追加(toks, 建("ARROW", "->"))。
    设 p 为 p 加 2。
  否则：
    列表追加(toks, 建("SUB", "减"))。
    设 p 为 p 加 1。
  结束。
  设 已处理 为 真。
结束。
# ... 类似处理其他运算符
```

## 任务 2：语法分析器 — 纯缩进支持（移除 `结束` 依赖）

- [ ] **步骤 2.1：修改 `compile_block` 函数**

当前 `compile_block` 使用 `find_matching_end` 和 `结束` 关键字判断块边界。改为使用 INDENT/DEDENT token 判断：
- 遇到 DEDENT token 时，退出当前块（返回 out）
- 不需要 `结束` 关键字
- 保留 `结束` 的兼容处理（如果遇到 `结束`，同样退出）

- [ ] **步骤 2.2：修改 `comp_if` 函数**

当前 `comp_if` 调用 `find_matching_end` 寻找 `结束`。改为直接编译到遇到 DEDENT 或 `否则`/`否则如果`。同样 INDENT token 标记块开始。

- [ ] **步骤 2.3：修改 `comp_while`、`comp_for`、`compile_class`**

类似地移除对 `find_matching_end` 和 `结束` 的依赖，使用 INDENT/DEDENT 判断块边界。

- [ ] **步骤 2.4：修改 `comp_try` 函数**

`comp_try` 也需要使用缩进层级来判断 `捕获`/`最终` 块的边界。

- [ ] **步骤 2.5：修改 `compile_stmts` 函数**

`compile_stmts` 是语句级编译函数。修改使其在遇到 DEDENT 时停止编译（跳过已处理的 token）。

- [ ] **步骤 2.6：支持单行块**

修改 `如果`、`当`、`遍历` 等语句的处理：`：` 后如果直接跟着语句（无换行），则直接编译该语句，不等待缩进块。

- [ ] **步骤 2.7：运行纯缩进测试**

创建测试验证：
```duan
段测试：
  如果真输出("是")
否则：
  输出("否")
```

## 任务 3：语法分析器 — 类型注解支持

当前 `comp_set` 处理 `设x为10`。需要扩展支持：
