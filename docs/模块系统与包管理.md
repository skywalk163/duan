# 段言模块系统与包管理

## 概述

段言实现了基于文件的模块化系统，并通过 `package.toml` 配置文件提供包管理功能。模块是编译和符号隔离的基本单位，包是依赖管理和分发的基本单位。

**核心特性**：
- 📦 **文件即模块**：每个 `.duan` 文件是一个独立模块，有独立符号表
- 🔁 **依赖图拓扑排序**：自动分析模块依赖关系，按正确顺序编译
- ⚠️ **循环依赖检测**：发现模块间的循环引用并给出清晰错误
- 🎯 **`package.toml` 配置**：声明包名、版本、依赖、入口文件
- 🔗 **`导入` 语句**：显式导入其他模块的段落、类、接口
- 🌐 **模块作用域隔离**：模块内定义默认私有，需显式导出

## 模块系统

### 什么是模块

每个 `.duan` 文件编译为一个独立模块，拥有独立的符号表。

```
项目/
├─ main.duan        ← 主模块，程序入口
├─ utils.duan       ← 工具模块
├─ math/
│   ├─ basic.duan   ← 数学基础模块
│   └─ advanced.duan ← 高级数学模块
└─ package.toml
```

### 模块编译流程

```
1. 扫描项目目录，收集所有 .duan 文件
2. 解析每个文件的 `导入` 语句，构建依赖图
3. 拓扑排序，确定编译顺序
4. 逐个模块编译：
   ├─ 词法分析（tokens）
   ├─ 语法分析（AST）
   ├─ HM 类型推断（两阶段）
   └─ 代码生成（Python）
5. 合并生成的 Python 代码，添加运行时函数
6. 输出最终可执行的 .py 文件
```

### 模块依赖图

```
                ┌─────────────┐
                │  advanced.  │
                │    数学    │
                └──┬───────┬──┘
                   │       │
                   ▼       ▼
            ┌─────────┐  ┌──────────┐
            │  basic  │  │  utils   │
            │  数学   │  │          │
            └────┬────┘  └────┬─────┘
                 │             │
                 ▼             ▼
                 ┌──────────────┐
                 │   main.duan  │
                 │  (程序入口)  │
                 └──────────────┘

编译顺序: basic, utils, advanced, main
```

### 拓扑排序算法

1. 构建邻接表：`{模块A: [依赖模块1, 依赖模块2, ...]}`
2. 计算入度：每个模块被多少模块依赖
3. 选择入度为 0 的模块加入队列
4. 逐个编译，更新依赖它的模块的入度
5. 若队列空但仍有模块未编译 → **循环依赖**

```python
# 简化实现
def topo_sort(graph):
    in_degree = {node: 0 for node in graph}
    for node, deps in graph.items():
        for dep in deps:
            in_degree[node] += 1  # node 依赖 dep，node 的入度+1

    queue = [n for n in in_degree if in_degree[n] == 0]
    result = []
    while queue:
        node = queue.pop()
        result.append(node)
        # 减少依赖于 node 的模块的入度
        for other, deps in graph.items():
            if node in deps:
                in_degree[other] -= 1
                if in_degree[other] == 0:
                    queue.append(other)
    return result  # 若长度 != len(graph)，存在环
```

### 循环依赖检测

```
模块 A 导入 B
模块 B 导入 A

错误: 检测到循环依赖: A → B → A
建议: 将共享代码提取到模块 C，A 和 B 都导入 C
```

### 导入语法

```段言
# 导入整个模块
导入 utils。

# 使用模块内容
utils.工具()
utils.辅助()

# 导入特定名称
从 utils 导入 工具, 辅助。

# 直接使用
工具()
辅助()

# 跨目录导入
从 math.basic 导入 加法, 乘法。
加法(三, 五)
```

### 模块作用域规则

1. **默认私有**：模块内定义的名称在模块外不可见
2. **显式导出**（未来）：通过 `导出 名称。` 或配置文件声明导出
3. **命名空间**：`模块.名称` 访问，避免名称冲突
4. **HM 推断跨模块**：导入的段的类型签名被保留，可在当前模块继续推断

## 包管理系统（package.toml）

### 配置文件格式

```toml
[package]
name = "我的应用"
version = "0.1.0"
description = "段言示例项目"
authors = ["开发者 <dev@example.com>"]
entry = "main.duan"          # 程序入口文件
language_version = "1.0"    # 段言语言版本

[dependencies]
# 本地依赖（相对路径）
"工具库" = { path = "../toolkit" }

# Git 依赖（未来版本）
# "网络库" = { git = "https://github.com/example/net.git", tag = "v1.0" }

# 注册表依赖（未来版本）
# "数学库" = "1.2"

[build]
output = "output.py"        # 编译输出文件名
target = "python3"          # 目标运行时
optimize = true             # 是否启用优化
```

### PackageConfig 数据结构

```python
@dataclass
class PackageConfig:
    name: str = "unnamed"
    version: str = "0.1.0"
    description: str = ""
    authors: list = None
    entry: str = "main.duan"
    dependencies: dict = None
    build_output: str = "output.py"
    build_target: str = "python3"
    build_optimize: bool = True

    @classmethod
    def from_toml(cls, path: str) -> 'PackageConfig':
        """从 TOML 文件加载配置"""
        with open(path, 'r', encoding='utf-8') as f:
            data = tomllib.loads(f.read())
        return cls._from_dict(data)

    @classmethod
    def from_dict(cls, data: dict) -> 'PackageConfig':
        """从字典加载配置"""
        pkg = data.get('package', {})
        deps = data.get('dependencies', {})
        build = data.get('build', {})
        return cls(
            name=pkg.get('name', 'unnamed'),
            version=pkg.get('version', '0.1.0'),
            description=pkg.get('description', ''),
            authors=pkg.get('authors', []),
            entry=pkg.get('entry', 'main.duan'),
            dependencies={k: v for k, v in deps.items()},
            build_output=build.get('output', 'output.py'),
            build_target=build.get('target', 'python3'),
            build_optimize=build.get('optimize', True),
        )
```

```bash
$ cd 我的项目

# 或
$ python src/compiler.py build
```

编译流程：

```
1. 加载 package.toml → PackageConfig
2. 发现所有 .duan 文件 → 模块列表
3. 解析每个模块的 `导入` 语句 → 依赖边
4. 构建依赖图 → {模块: [依赖模块]}
5. 拓扑排序 → 编译顺序
6. 按顺序编译每个模块:
   - 词法分析 → tokens
   - 语法分析 → AST (Module)
   - HM 类型推断 (预扫描 → 段体推断 → 泛化)
   - 代码生成 → Python 代码片段
7. 合并所有模块的 Python 代码
8. 添加运行时函数 (_duan_unwrap, _duan_assert_not_none 等)
9. 写入 output.py
10. 完成 ✓
```

### 跨模块类型推断交互

```
模块 A (utils.duan):
    段落 加(甲, 乙):
        返回 甲 加 乙。
    结束。
    ↓ 编译后
    加: (数, 数) -> 数    ← 保存在模块符号表中

模块 B (main.duan):
    从 utils 导入 加。
    定义 结果 等于 加(三, 五)。
    ↓ 推断
    加 的类型签名来自模块 A
    结果: 数   ← 跨模块推断成功
```

## 依赖管理的三种形式

### 1. 本地依赖（path）

```toml
[dependencies]
"我的工具库" = { path = "../tools" }
```

编译器会：
1. 查找 `../tools/package.toml`
2. 编译那个项目
3. 将其模块加入当前项目的导入搜索路径

### 2. Git 依赖（未来版本）


### 3. 注册表依赖（未来版本）

## 与 HM 推断和可空类型的协同

### 跨模块 HM 推断

```
模块 A:
  段落 获取值():      ← 推断为 () -> 数
      返回 四十二。
  结束。

模块 B:
  从 A 导入 获取值。
  定义 x 等于 获取值()。  ← x: 数  ✓ 跨模块推断成功
```

### 跨模块可空类型

```
模块 A:
  段落 可能空():        ← 推断为 () -> 数|空
      如果 随机 大于 五:
          返回 空。
      否则:
          返回 五。
      结束。
  结束。

模块 B:
  从 A 导入 可能空。
  定义 值 等于 可能空()。  ← 值: 数|空
  显示 值!。             ← 需要解包 ✓ 检查正确
```

## 错误处理

| 错误类型 | 场景 | 处理 |
|---|---|---|
| **模块未找到** | `导入 不存在。` | 报错并列出可用模块 |
| **名称未导出** | `从 A 导入 私有段。` | 报错并列出可导出名称 |
| **循环依赖** | A→B→A | 报错并显示环路径 |
| **配置错误** | package.toml 格式错误 | 报错并指示行号 |
| **入口文件不存在** | entry 指定的文件找不到 | 报错 |

## 未来路线图

1. **版本号约束**：`>=1.0, <2.0` 语义化版本
2. **锁文件**：`package.lock` 固定依赖版本，保证可复现构建
3. **中央注册表**：`段言 install 包名` 从远程安装
4. **增量编译**：只重新编译变更的模块
5. **并行编译**：无依赖的模块并行编译
6. **模块测试**：每个模块可以有自己的 `__test__.duan`
7. **条件编译**：根据平台/特性标志选择性编译
